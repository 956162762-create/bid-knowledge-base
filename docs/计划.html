设计原则

# 原则 说明
1 结构化为主，RAG 兜底 条款号/表格 O(1) 命中；语义搜索只用于模糊自然语言
2 应用层从第一天就定义 技术标制作不是事后补丁，是整个系统的目标出口
3 API 层零外部依赖 基于 NiceGUI 自带 FastAPI，不部署新服务
4 信息完整性在检索阶段保证 表格完整存储，不指望 LLM 拼碎片
5 LLM 只做三件事 阅读理解 · 格式化输出 · 按模板生成文本

应用范围

场景 说明 优先级
信息提取 从招标文件精准提取条款、表格、要求、评分标准 最高
技术标辅助 生成技术标大纲 + 逐项响应模板 + docx 导出 最高
商务标辅助 报价相关（后续扩展） 低
法律推理 明确不做 ✗

LLM 能力： 千问 32B 在结构化素材完备的前提下完全够用。不必在法律推理或 LLM 选型上花时间。

3. 核心层 — 招标知识库引擎

核心层的每个模块都是独立的 Python 类，API 层直接调用，不依赖 UI。

3.1 结构化解析引擎（主引擎 · Phase 0a 实现）

职责： 把 docx 拆成结构化数据，支撑所有精确查询。

五大产出：

产出 存储 查询方式 应用场景
条款树 clause_nodes 表 条款号 → O(1)；路径遍历 "§6.1.16 是什么"
表格库 tables 表 表格编号/标题 → O(1) "危大工程清单" → 完整 A-8 表
交叉引用索引 cross_references 表 正向/反向追踪 "A-8 和 §6.1.16 的关系"
条款号哈希 内存 dict + SQLite dict["§6.1.16"] < 1ms 精确条款定位
文档层级视图 clause_nodes.path 按路径遍历 "第六章有哪些内容"

核心 API：

class StructuredEngine:
    def get_clause(self, number: str) -> Optional[ClauseNode]: ...
    def get_clause_tree(self, chapter: str) -> List[ClauseNode]: ...
    def get_table(self, number: str) -> Optional[Table]: ...
    def search_tables(self, keyword: str) -> List[Table]: ...
    def get_xrefs_from(self, number: str) -> List[CrossReference]: ...
    def get_document_hierarchy(self) -> DocumentHierarchy: ...
    def search_clauses_by_keyword(self, kw: str) -> List[ClauseNode]: ...

数据库 Schema：

-- 条款树
CREATE TABLE clause_nodes (
    id          INTEGER PRIMARY KEY,
    parent_id   INTEGER REFERENCES clause_nodes(id),
    node_type   TEXT CHECK(node_type IN ('volume','part','chapter','section','clause')),
    number      TEXT,            -- "§6.1.16"
    title       TEXT,
    content     TEXT,
    level       INTEGER,
    page_start  INTEGER,
    path        TEXT             -- "/第二卷/第六章/§6.1.16"
);
CREATE INDEX idx_clause_number ON clause_nodes(number);

-- 表格库
CREATE TABLE tables (
    id          INTEGER PRIMARY KEY,
    table_number TEXT,           -- "A-8"
    title       TEXT,
    chapter_ref TEXT,            -- "§6.1.16"
    page_number INTEGER,
    structure   TEXT,            -- JSON：行列结构
    merged_cells TEXT,           -- JSON：合并单元格
    raw_html    TEXT,            -- 完整 HTML 表格
    checkbox_states TEXT,        -- JSON：{"row_col": "checked|unchecked"}
    row_count   INTEGER,
    col_count   INTEGER
);

-- 条款号哈希索引（O(1) 查找）
CREATE TABLE clause_number_index (
    number      TEXT PRIMARY KEY,
    entity_type TEXT CHECK(entity_type IN ('clause','table')),
    entity_id   INTEGER
);

-- 交叉引用（Phase 0b）
CREATE TABLE cross_references (
    id              INTEGER PRIMARY KEY,
    from_clause_id  INTEGER REFERENCES clause_nodes(id),
    from_table_id   INTEGER REFERENCES tables(id),
    to_clause_id    INTEGER REFERENCES clause_nodes(id),
    to_table_id     INTEGER REFERENCES tables(id),
    to_number       TEXT,
    xref_type       TEXT CHECK(xref_type IN ('see','refer_to','according_to','appendix'))
);

3.4 规则引擎（Phase 2）

三类管道钩子：

pre-retrieval   → 修改搜索策略（如"危大工程 → 同时查 A-8 + §6.1.16"）
post-retrieval  → 过滤/增强检索结果
post-generation → 格式化答案（来源标注、符号统一）

规则 JSON 示例：

{
  "name": "危大工程-返回完整表格",
  "rule_type": "return_table",
  "condition": {"keywords": ["危大工程", "危险性较大"]},
  "action": {
    "table_numbers": ["A-8"],
    "also_include_clauses": ["§6.1.16"],
    "xref_depth": 1
  }
}
规则动作 说明 示例
return_table 直接返回指定完整表格 "危大工程清单" → A-8
return_clause 直接返回指定条款 "§6.1.16" → O(1)
source_label 自动标注出处 "来自合同文件 §6.1.16"
checkbox_format 统一勾选符号 [x] → [✓]
include_always 强制包含指定内容 回答必须含某条款/表格

3.5 智能 Agent（Phase 4）

职责： 编排结构化引擎的能力，处理跨模块查询。纯 Python 类，不引入额外 LLM 成本。

class QueryPlanner:
    def plan(self, query: str) -> QueryPlan:
        # 意图分类（LLM 辅助 + 关键词规则兜底）
        intent = self.classifier.classify(query)
        # 按意图路由到结构化引擎的不同方法
        match intent:
            case CLAUSE_LOOKUP → clause_number_index.get(number)
            case TABLE_QUERY  → table_store.search(keywords)
            case XREF_TRACE   → xref_index.trace(number)
            case HIERARCHY    → clause_tree.walk(chapter)
            case SEMANTIC     → vector_store.search(query)  # 兜底

意图分类器： LLM 快速分类为主（< 1s），纯关键词规则兜底。处理边界情况如"6.1.16条"（无§符但是条款查询）、"危大工程有哪些"（无"清单"词但是查表格）。

4.1 端点定义

# api/routes.py
from nicegui import app as ng_app

# ── 文档分析 ──

@ng_app.api.post("/api/projects/{project_id}/analyze")
async def analyze_bid_doc(project_id: str, file: UploadFile):
    """上传招标文件 → 结构化解析 → 返回分析摘要"""
    result = await ingest_service.ingest_and_analyze(project_id, file)
    return {
        "document_id": result.doc_id,
        "clause_count": result.clause_count,
        "table_count": result.table_count,
        "key_requirements": result.requirements[:20],  # 前 20 条关键要求
        "scoring_items": result.scoring_items,
        "structure_map": result.hierarchy.to_dict()
    }

# ── 智能问答 ──

@ng_app.api.post("/api/projects/{project_id}/query")
async def query(project_id: str, body: QueryRequest):
    """自然语言查询招标文件"""
    result = await query_service.query(project_id, body.question)
    return {
        "answer": result.answer,
        "sources": [s.to_dict() for s in result.sources],
        "intent": result.intent,
        "path": result.path,           # "structured" | "semantic"
        "applied_rules": result.applied_rules
    }

# ── 投标策略建议 ──

@ng_app.api.get("/api/projects/{project_id}/suggest")
async def suggest_strategy(project_id: str):
    """基于招标文件分析，返回投标策略建议"""
    reqs = await requirement_extractor.extract_all(project_id)
    return {
        "must_respond": reqs.filter(type="must"),

    }

    }

# ── 项目信息 ──

    """获取项目关键信息提取结果"""

# api/schemas.py
from pydantic import BaseModel

class QueryRequest(BaseModel):
    question: str
    stream: bool = True
    use_agent: bool = False

class QueryResponse(BaseModel):
    answer: str
    sources: List[SourceRef]
    intent: str
    path: str                          # "structured" | "semantic"
    applied_rules: List[str]

class SourceRef(BaseModel):
    clause_number: Optional[str]       # "§6.1.16"
    table_number: Optional[str]        # "A-8"
    volume: Optional[str]
    chapter: Optional[str]
    content_preview: str

class ExportRequest(BaseModel):
    sections: List[str]                # 要导出的技术标章节
    format: str = "docx"

5.2 核心数据结构

@dataclass
class RequirementItem:
    """从招标文件提取的单条要求"""
    clause_ref: str              # "§6.1.16"
    requirement_type: str        # "must" | "should" | "scoring" | "attachment"
    content: str                 # 要求原文
    score_weight: Optional[float]

@dataclass
class ComplianceMatrix:
    """投标响应对照表"""
    items: List[RequirementItem]
    total_score: float
    must_count: int
    coverage_pct: float          # 已响应 / 总要求

@dataclass
class TechBidOutline:
    """技术标大纲"""
    sections: List[OutlineSection]
    compliance: ComplianceMatrix

8. 关键技术决策

决策点 选择 理由
架构分层 四层：应用 → API → 核心 → 数据 应用层独立于引擎，API 可被多个前端调用
检索范式 结构化为主，RAG 兜底 条款号/表格 O(1)；语义仅用于模糊查询
API 框架 NiceGUI 自带 FastAPI 零额外服务，UI 和 API 共享同一进程
主存储 SQLite（结构化） 条款树/表格库/索引用关系表
向量存储 ChromaDB（辅助） 仅叙述性段落入向量库
条款号查找 内存哈希 + SQLite 持久化 O(1) < 1ms
表格存储 完整 JSON 结构 不切片，勾选框/合并单元格完整保留
LLM Ollama qwen3:32b 结构化素材完备，32B 完全够用
意图分类 LLM 快速分类 + 关键词兜底 纯正则无法覆盖"危大工程有哪些"等边界
状态管理 NiceGUI reactive state 框架原生
项目隔离 每项目独立 SQLite + ChromaDB 文件夹 删除=删文件夹，备份=复制文件夹
版本管理 documents 表 version + parent_doc_id V1 预留字段，V2 实现版本链 UI
法律推理 不做 专注信息提取 + 技术标制作
向后兼容 保留 main.py CLI 脚本化场景
